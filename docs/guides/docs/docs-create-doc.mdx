# 문서 만들기

`greeting.md`라는 Markdown 파일을 생성하고 `docs` 디렉토리 아래에 배치하세요.

```bash
website # 사이트의 루트 디렉토리
├── docs
│   └── greeting.md
├── src
│   └── pages
├── docusaurus.config.js
├── ...
```

```md
---
description: 풍부한 콘텐츠가 포함된 문서 페이지를 만듭니다.
---

# Docusaurus에서 안녕하세요

오픈 소스 프로젝트를 위한 문서 사이트를 만들 준비가 되셨나요?

## 헤더

오른쪽 상단의 목차에 표시됩니다

사용자가 스크롤을 내리지 않거나 많이 읽지 않아도 이 페이지가 무엇에 관한 것인지 알 수 있도록 합니다.

## 기본적으로 h2와 h3만 목차에 포함됩니다.

문서별로 또는 테마 구성에서 목차 제목 수준을 구성할 수 있습니다.

헤더는 계층 구조가 명확하도록 잘 간격을 두고 있습니다.

- 리스트는 도움이 됩니다
- 사용자가 기억하기를 원하는
- 핵심 포인트를 제시하는 데
  - 그리고 중첩할 수 있습니다
    - 여러 번
```

:::note

`docs` 디렉토리 아래의 밑줄(`_`)로 시작하는 모든 파일은 "부분" 페이지로 취급되며 기본적으로 무시됩니다.

[부분 페이지 가져오기](../markdown-features/markdown-features-react.mdx#importing-markdown)에 대해 자세히 알아보세요.

:::

## 문서 프론트매터 {#doc-front-matter}

[프론트매터](../markdown-features/markdown-features-intro.mdx#front-matter)는 문서 페이지에 추가 메타데이터를 제공하는 데 사용됩니다. 프론트매터는 선택 사항입니다. Docusaurus는 프론트매터 없이도 모든 필요한 메타데이터를 추론할 수 있습니다. 예를 들어, 아래에서 소개하는 [문서 태그](#doc-tags) 기능은 프론트매터 사용을 요구합니다. 가능한 모든 필드에 대해서는 [API 문서](../../api/plugins/plugin-content-docs.mdx#markdown-front-matter)를 참조하세요.

## 문서 태그 {#doc-tags}

태그는 프론트매터에서 선언되며 [문서 사이드바](./sidebar/index.mdx) 외에 또 다른 차원의 분류를 도입합니다.

태그를 인라인으로 정의하거나 [`tags 파일`](../../api/plugins/plugin-content-docs.mdx#tags-file)(선택 사항, 일반적으로 `docs/tags.yml`)에서 선언된 미리 정의된 태그를 참조할 수 있습니다.

다음 예시에서:

- `docusaurus`는 `docs/tags.yml`에 선언된 미리 정의된 태그 키를 참조합니다
- `Releases`는 `docs/tags.yml`에 존재하지 않기 때문에 인라인 태그입니다

```md title="docs/my-doc.md"
---
tags:
  - Releases
  - docusaurus
---

# 제목

내용
```

```yml title="docs/tags.yml"
docusaurus:
  label: 'Docusaurus'
  permalink: '/docusaurus'
  description: 'Docusaurus 프레임워크와 관련된 문서'
```

:::tip

태그는 `tags: [Demo, Getting started]`와 같이 선언할 수도 있습니다.

가능한 모든 [Yaml 배열 구문](https://www.w3schools.io/file/yaml-arrays/)에 대해 자세히 알아보세요.

:::

## 폴더 구조 구성하기 {#organizing-folder-structure}

`docs` 폴더 아래의 Markdown 파일 배열은 Docusaurus 콘텐츠 생성에 여러 영향을 미칠 수 있습니다. 그러나 대부분은 파일 구조와 분리될 수 있습니다.

### 문서 ID {#document-id}

모든 문서는 고유한 `id`를 가집니다. 기본적으로 문서 `id`는 루트 docs 디렉토리를 기준으로 한 문서 이름(확장자 제외)입니다.

예를 들어, `greeting.md`의 ID는 `greeting`이고, `guide/hello.md`의 ID는 `guide/hello`입니다.

```bash
website # 사이트의 루트 디렉토리
└── docs
   ├── greeting.md
   └── guide
      └── hello.md
```

그러나 `id`의 **마지막 부분**은 프론트매터에서 사용자가 정의할 수 있습니다. 예를 들어, `guide/hello.md`의 내용이 아래와 같이 정의되면 최종 `id`는 `guide/part1`입니다.

```md
---
id: part1
---

Lorem ipsum
```

ID는 사이드바를 수동으로 작성하거나 문서 관련 레이아웃 컴포넌트 또는 훅을 사용할 때 문서를 참조하는 데 사용됩니다.

### 문서 URL {#doc-urls}

기본적으로 문서의 URL 위치는 `docs` 폴더를 기준으로 한 파일 경로입니다. 단, 파일 이름이 다음 중 하나인 경우 파일 이름이 URL에 포함되지 않습니다:

- `index`로 명명된 경우 (대소문자 구분 없음): `docs/Guides/index.md`
- `README`로 명명된 경우 (대소문자 구분 없음): `docs/Guides/README.mdx`
- 부모 폴더와 같은 이름인 경우: `docs/Guides/Guides.md`

모든 경우에 기본 슬러그는 `/index`, `/README` 또는 중복 `/Guides` 세그먼트 없이 `/Guides`만 됩니다.

:::note

이 규칙은 [카테고리 인덱스 규칙](./sidebar/autogenerated.mdx#category-index-convention)과 정확히 동일합니다. 그러나 `isCategoryIndex` 구성은 문서 URL에 영향을 미치지 _않습니다_.

:::

문서의 URL을 변경하려면 `slug` 프론트매터를 사용하세요.

예를 들어, 사이트 구조가 다음과 같다고 가정해 봅시다:

```bash
website # 사이트의 루트 디렉토리
└── docs
    └── guide
        └── hello.md
```

기본적으로 `hello.md`는 `/docs/guide/hello`에서 사용할 수 있습니다. URL 위치를 `/docs/bonjour`로 변경할 수 있습니다:

```md
---
slug: /bonjour
---

Lorem ipsum
```

`slug`는 문서 플러그인의 `routeBasePath`(기본값은 `/docs`)에 추가됩니다. URL에서 `/docs` 부분을 제거하는 방법은 [문서 전용 모드](docs-introduction.mdx#docs-only-mode)를 참조하세요.

:::note

다음과 같이 사용할 수 있습니다:

- 절대 슬러그: `slug: /mySlug`, `slug: /`...
- 상대 슬러그: `slug: mySlug`, `slug: ./../mySlug`...

:::

문서를 루트에서 사용할 수 있게 하고 `https://docusaurus.io/docs/`와 같은 경로를 가지려면 슬러그 프론트매터를 사용할 수 있습니다:

```md
---
id: my-home-doc
slug: /
---

Lorem ipsum
```

### 사이드바 {#sidebars}

[자동 생성 사이드바](./sidebar/autogenerated.mdx)를 사용할 때 파일 구조가 사이드바 구조를 결정합니다.

파일 시스템 구성에 대한 우리의 권장 사항은 다음과 같습니다: 파일 시스템을 사이드바 구조와 일치시키고(그래서 `sidebars.js` 파일을 수동으로 작성할 필요가 없습니다), 각 문서의 URL을 사용자 정의하려면 `slug` 프론트매터를 사용하세요.
